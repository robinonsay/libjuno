= Log Module

== Purpose

The Log module (`log_api.h`) provides a minimal, pluggable logging facade for embedded targets.
It defines four severity levels (Debug, Info, Warning, Error) as vtable functions.
Implementations decide the sink (UART, RTT, buffer) and formatting strategy.

== Data Structures

=== Log Root (`JUNO_LOG_ROOT_T`)

[source,c]
----
struct JUNO_LOG_ROOT_TAG JUNO_MODULE_ROOT(JUNO_LOG_API_T, JUNO_MODULE_EMPTY);
----

=== Log API (`JUNO_LOG_API_T`)

[cols="1,3"]
|===
| Function | Description

| `LogDebug(ptJunoLog, pcMsg, ...)` | Log a debug-level message (most verbose).
| `LogInfo(ptJunoLog, pcMsg, ...)` | Log an info-level message.
| `LogWarning(ptJunoLog, pcMsg, ...)` | Log a warning-level message.
| `LogError(ptJunoLog, pcMsg, ...)` | Log an error-level message.
|===

All functions accept variadic arguments (printf-like format).

== Interface Design

`JunoLog_LogInit(ptLog, ptApi, pfcnFailureHandler, pvUserData)`::
Inline function that wires the log API and failure handler into the root.
Returns `JUNO_STATUS_SUCCESS` on success; `JUNO_STATUS_NULLPTR_ERROR` if `ptLog` is NULL.

== Algorithm Descriptions

No algorithms — the Log module defines the interface only.
Formatting and output are implementation-specific.

== Error Handling

* `JunoLog_LogInit` validates the log pointer via `JUNO_ASSERT_EXISTS`.
* All log functions return `JUNO_STATUS_T` for consistent error propagation.
* The module root carries a failure handler for self-diagnosis.
* Format string interpretation is implementation-defined; callers should avoid undefined specifiers in freestanding builds.

== Design Rationale

The logging interface is minimal (four severity levels, printf-like format) to remain portable and lightweight.
No filtering, buffering, or thread-safety guarantees are imposed at the interface level — these are implementation concerns that vary drastically between targets.
