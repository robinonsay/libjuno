== Introduction

=== Purpose

This Software Design Document (SDD) describes the internal design of LibJuno version 1.0.1, a freestanding C11 embedded systems micro-framework.
It follows the IEEE 1016 standard structure and is intended for developers, integrators, reviewers, and certification auditors who need to understand the design decisions, data structures, interfaces, and algorithms within the library.

=== Scope

This SDD covers all modules of LibJuno:

* Core infrastructure: Status, Module System, Macros, Common Types
* Memory subsystem: Pointer, Memory Allocator, Block Allocator
* Data structures: Array, Heap, Queue, Stack, Map
* Algorithms: CRC, Hash, Math (vectors, quaternions)
* System services: Time, IO (Async, I2C, SPI), Log, Mutex
* Frameworks: State Machine, Software Bus Broker, Application, Scheduler

== System Architecture

=== Architectural Overview

LibJuno is a dependency-injection-oriented C11 micro-framework for embedded systems.
Its architecture is built on four foundational principles:

. **Zero dynamic allocation** â€” all memory is caller-owned and injected.
. **Freestanding compatibility** â€” the library compiles with `-nostdlib -ffreestanding`.
. **Vtable-based dependency injection** â€” every module uses function-pointer tables for polymorphic dispatch.
. **Explicit error propagation** â€” all fallible functions return status codes; no exceptions, no `errno`, no `longjmp`.

The library provides reusable building blocks (data structures, memory management, CRC, math, time, IO abstractions) and a lightweight application framework (app lifecycle, state machine, software bus, scheduler) that integrators compose into their system.

=== Design Rationale: Why Vtable-Based DI

LibJuno chose vtable-based dependency injection over alternative C polymorphism approaches (switch-based dispatch, `_Generic`, per-instance function pointers) for the following reasons:

**Architectural decoupling.** A root object carrying a `const API_T *ptApi` pointer supports passing modules behind a stable root type.
Dependencies can be injected explicitly, and concrete implementations can be swapped without central registries or framework-owned dispatch code.

**Why not switch-based dispatch.** A switch-based approach centralizes polymorphism around a type tag, coupling the common interface layer to every concrete implementation.
Each new backend requires editing the switch logic and shared dispatch functions â€” the opposite of LibJuno's goal of making components easy to adapt, share, and compose.
With the vtable pattern, a new implementation defines a new derived type plus a new static API table, without requiring edits to a central dispatcher.

**Vtable indirection vs. switch dispatch.** A vtable call adds one pointer dereference and typically prevents cross-interface inlining.
A switch-based dispatcher can sometimes be inlined more aggressively.
LibJuno favors architectural decoupling, substitutability, and test injection over dispatch micro-optimization â€” consistent with its emphasis on portability, explicit interfaces, and reusable building blocks.

**Shared vtable per type vs. per-instance function pointers.** LibJuno stores `const API_T *ptApi` in the root â€” one pointer per instance, with the function table shared by all instances of the same implementation.
This is smaller and more consistent than per-instance function pointers, especially for embedded targets with many instances sharing the same behavior.

**`const` vtable and ROM placement.** The module macros define the root field as `const API_T *ptApi`, and implementations provide `static const` API tables.
This supports placing vtables in read-only storage/flash on embedded systems, reducing RAM pressure and preventing accidental mutation.

**Why not `_Generic` or macro-driven generics.** `_Generic` is selected at compile time from the static type at the call site; it does not provide runtime substitution through a common root pointer.
LibJuno's module system is explicitly built around runtime polymorphism and injected dependencies, so `_Generic` would supplement type ergonomics, not replace the architecture.

=== Module System Pattern: Root / Derivation / Union

Every LibJuno module follows a three-part structural pattern:

[source,c]
----
// 1. Root â€” common interface fields
struct MY_ROOT_TAG JUNO_MODULE_ROOT(MY_API_T,
    // additional root-level state
);

// 2. Derivation â€” embeds root as first member
struct MY_IMPL_TAG JUNO_MODULE_DERIVE(MY_ROOT_T,
    // implementation-specific state
);

// 3. Union â€” sized for the largest variant
union MY_MODULE_T JUNO_MODULE(MY_API_T, MY_ROOT_T,
    MY_IMPL_T tImpl;
);
----

=== Design Rationale: Root/Derivation/Union Layering

**Why the root/derivation split.** API functions accept `ROOT_T *` â€” the common base type.
A flat struct per implementation would require every call site to know the concrete type (breaking abstraction) or use `void *` casts (breaking type safety).
Because `JUNO_MODULE_SUPER` is always the first member, a `DERIVATION_T *` can be safely cast to `ROOT_T *` by the C standard (Â§6.7.2.1Â¶15).

**Why the union.** The union serves stack allocation sizing with type erasure.
LibJuno requires zero dynamic allocation, so the caller must allocate the module instance â€” but may not know which derivation will be initialized at runtime.
The union is sized to the largest derivation.
It also provides C-standard type-punning safety (Â§6.5.2.3): accessing the root member is always valid because the root is a common initial sequence shared by all members.

**Why first-member embedding over `container_of`.** Three reasons:

. **No `offsetof` dependency** â€” the cast is `(ROOT_T *)ptDerived`, zero arithmetic, zero macros.
. **No ownership inversion** â€” root and derivation are physically contiguous in a single allocation.
. **Determinism and locality** â€” same cache line, no pointer indirection.

== Data Design

=== Common Types

LibJuno defines a small set of common value types used across all modules:

**`JUNO_STATUS_T`** (`int32_t`) â€” the unified error code type.
Zero indicates success; non-zero values indicate specific failure conditions.

**Result types** (`JUNO_MODULE_RESULT(NAME_T, OK_T)`) â€” structs combining `{JUNO_STATUS_T tStatus; OK_T tOk;}`.
Pre-defined specializations: `JUNO_RESULT_BOOL_T`, `JUNO_RESULT_UINT32_T`, `JUNO_RESULT_F64_T`, `JUNO_RESULT_SIZE_T`, `JUNO_RESULT_VOID_PTR_T`.

**Option types** (`JUNO_MODULE_OPTION(NAME_T, SOME_T)`) â€” structs combining `{bool bIsSome; SOME_T tSome;}`.
Used for operations where absence is a legitimate state (peek on empty collection, optional next-state in SM).

**`JUNO_POINTER_T`** â€” the fat pointer descriptor `{ptApi, pvAddr, zSize, zAlignment}`.
Carries self-describing type information and operations for type-safe memory access.

=== Memory Ownership Model

All memory in LibJuno is **caller-owned and injected**:

. The caller allocates storage (stack, static, or platform-specific memory pool).
. The caller passes storage to an `Init` function along with all dependencies.
. The module operates exclusively on the injected storage.
. The caller is responsible for the lifetime of all injected memory.

No module in LibJuno calls `malloc`, `calloc`, `realloc`, or `free`.
Memory pools (block allocator) operate on caller-provided backing buffers.

=== Design Rationale: Result/Option Types and Error Handling

**Why Rust-style Result/Option in C11.** `errno` is a hosted-only thread-local global unavailable in freestanding builds and violates the "no global mutable state" constraint.
Out-parameters separate the value from its validity â€” nothing prevents ignoring the status and using a garbage out-parameter.
Result structs bind the value and status in a single aggregate with `JUNO_ASSERT_OK` enforcing check-before-use.

**Why failure handlers are diagnostic-only.** Three reasons:

. **Predictable WCET** â€” if handlers could `longjmp` or abort, the control flow of every library function becomes unanalyzable.
. **No stack unwinding** â€” `longjmp`/`setjmp` is hosted-only; exceptions require RTTI. The only portable freestanding flow control is "check status, return status."
. **Separation of policy from mechanism** â€” the library decides what constitutes an error; the application decides what to do about it (log, blink LED, set fault flag).

**Why the ASSERT macro family.** Each macro maps to a specific domain of validation:

* `JUNO_ASSERT_EXISTS(ptr)` â†’ always returns `JUNO_STATUS_NULLPTR_ERROR` â€” correct specific status.
* `JUNO_ASSERT_SUCCESS(s, ...)` â†’ variadic action statements for ergonomic propagation.
* `JUNO_ASSERT_OK(result, ...)` â†’ unwraps Result types.
* `JUNO_ASSERT_SOME(opt, ...)` â†’ unwraps Option types.

Each macro name is self-documenting: it reads as a named proof obligation.

== Interface Design

=== Module Initialization Contracts

Every module follows the same initialization pattern:

. Caller allocates all storage.
. Caller calls `Module_Init(...)` passing the instance, all dependencies, and an optional failure handler.
. `Init` wires the vtable, stores dependencies, and calls `Verify`.
. `Verify` validates all pointers and dependencies are non-NULL.
. All subsequent public functions call `Verify` at entry.

[source,c]
----
JUNO_STATUS_T JunoDs_Heap_Init(
    JUNO_DS_HEAP_ROOT_T *ptHeap,
    const JUNO_DS_HEAP_POINTER_API_T *ptHeapPointerApi,
    JUNO_DS_ARRAY_ROOT_T *ptHeapArray,
    JUNO_FAILURE_HANDLER_T pfcnFailureHdlr,
    JUNO_USER_DATA_T *pvUserData
);
----

=== Vtable Dispatch Pattern

API dispatch follows this pattern:

[source,c]
----
// Caller invokes via vtable:
ptModule->ptApi->DoSomething(&ptModule->tRoot, args...);

// Or through the root directly:
JUNO_MODULE_GET_API(ptModule, ROOT_T)->DoSomething((ROOT_T *)ptModule, args...);
----

The vtable is `const` and typically `static` â€” enabling ROM placement.
Dispatch requires one pointer dereference through `ptApi`, then a function pointer call.

=== Trait System (JUNO_TRAIT_ROOT)

Traits are lightweight interfaces without failure handler fields:

[source,c]
----
struct JUNO_POINTER_TAG JUNO_TRAIT_ROOT(JUNO_POINTER_API_T,
    void *pvAddr;
    size_t zSize;
    size_t zAlignment;
);
----

=== Design Rationale: Pointer Trait System

**Why a fat pointer with a vtable.** `JUNO_POINTER_T` binds `{pvAddr, zSize, zAlignment, ptApi}` into a self-describing, self-validating value type.
In freestanding environments, `memcpy` is not guaranteed to exist; by putting `Copy` behind a vtable in the pointer, the element knows how to copy itself â€” data structures never need `memcpy`.

**Why Copy/Reset on the pointer, not per-data-structure.** One Copy/Reset implementation per element type, reused by every data structure (heap, queue, stack, map).
Data structures remain type-agnostic; adding a new element type requires only a new `JUNO_POINTER_API_T` implementation.

**Why `JUNO_TRAIT_ROOT` (no failure handler).** The pointer is a pass-by-value ephemeral descriptor, not a stateful module instance.
Adding failure handler fields (+16 bytes on 64-bit) would bloat every data structure call's stack frame.
Failure reporting belongs to the owning module.

=== Software Bus Integration Pattern

The broker provides single-threaded publish/subscribe message fan-out:

[source,c]
----
// Publisher:
ptBrokerApi->Publish(ptBroker, ENGINE_CMD_MID, tMsgPointer);

// Subscriber (in its own OnProcess):
ptQueueApi->Dequeue(&ptCmdPipe->tRoot, tReturnPointer);
----

=== Design Rationale: Single-Threaded Broker

**Why single-threaded scope.** IPC mechanisms are platform-specific and non-abstractable without losing the properties that make them useful (blocking semantics, copy vs. zero-copy, priority inheritance, WCET guarantees).
The broker solves intra-thread fan-out portably; inter-thread transport belongs to the integrator.

**Why pipes (queues) over callbacks.** Three reasons:

. **Temporal decoupling** â€” Publish does O(1) enqueue; subscriber processes on its own schedule.
. **Bounded backpressure** â€” each pipe has fixed capacity; overflow returns an error status.
. **Subscriber control** â€” subscribers inspect, batch, and prioritize message consumption.

**Multi-threaded integration pattern.** One broker per thread, bridged by a user-written bridge app that forwards messages between brokers via platform IPC.

=== Scheduler Integration Pattern

The scheduler is a table-driven cyclic executive:

[source,c]
----
JUNO_SCH_TABLE_NEW(ptTable, 2, 3,
    &tAppA.tRoot, &tAppB.tRoot,   // Minor frame 0
    &tAppA.tRoot, NULL,            // Minor frame 1
    &tAppC.tRoot, &tAppB.tRoot,   // Minor frame 2
);
----

=== Design Rationale: Cyclic Executive

**Why static cyclic executive.** Deterministic timing by construction â€” execution order and boundaries are fixed at compile time.
No OS dependency (works on bare metal with a single timer).
Certifiable by inspection (DO-178C, MISRA, IEC 61508).
Complements RTOS preemption rather than replacing it.

**Why flattened 2D array.** Zero allocation (single caller-owned buffer), cache-friendly sequential access, O(1) index arithmetic, NULL slots for variable-length frames.

**Why injected Time module.** Clock hardware (`Now`) and blocking primitives (`Sleep`, `SleepTo`) are platform-specific.
Time math (add, subtract, conversions) is universal and provided by LibJuno.
The split follows DI: inject what varies, provide what's universal.
