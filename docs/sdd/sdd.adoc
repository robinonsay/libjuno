== Introduction

=== Purpose

This Software Design Document (SDD) describes the internal design of LibJuno version 1.0.1, a freestanding C11 embedded systems micro-framework.
It follows the IEEE 1016 standard structure and is intended for developers, integrators, reviewers, and certification auditors who need to understand the design decisions, data structures, interfaces, and algorithms within the library.

=== Scope

This SDD covers all modules of LibJuno:

* Core infrastructure: Status, Module System, Macros, Common Types
* Memory subsystem: Pointer, Memory Allocator, Block Allocator
* Data structures: Array, Heap, Queue, Stack, Map
* Algorithms: CRC, Hash, Math (vectors, quaternions)
* System services: Time, IO (Async, I2C, SPI), Log, Mutex
* Frameworks: State Machine, Software Bus Broker, Application, Scheduler

NOTE: LibJuno also provides C++11 template wrappers for the Queue and Stack data structures (`buff_api.hpp`, `juno_buff.hpp`).
These wrappers are thin convenience layers over the C API and are not covered in this SDD.
Refer to the Doxygen API reference for their interface documentation.

== System Architecture

=== Architectural Overview

LibJuno is a dependency-injection-oriented C11 micro-framework for embedded systems.
Its architecture is built on four foundational principles:

. **Zero dynamic allocation** — all memory is caller-owned and injected.
. **Freestanding compatibility** — the library compiles with `-nostdlib -ffreestanding`.
. **Vtable-based dependency injection** — every module uses function-pointer tables for polymorphic dispatch.
. **Explicit error propagation** — all fallible functions return status codes; no exceptions, no `errno`, no `longjmp`.

The library provides reusable building blocks (data structures, memory management, CRC, math, time, IO abstractions) and a lightweight application framework (app lifecycle, state machine, software bus, scheduler) that integrators compose into their system.

=== Design Rationale: Why Vtable-Based DI

LibJuno chose vtable-based dependency injection over alternative C polymorphism approaches (switch-based dispatch, `_Generic`, per-instance function pointers) for the following reasons:

**Architectural decoupling.** A root object carrying a `const API_T *ptApi` pointer supports passing modules behind a stable root type.
Dependencies can be injected explicitly, and concrete implementations can be swapped without central registries or framework-owned dispatch code.

**Why not switch-based dispatch.** A switch-based approach centralizes polymorphism around a type tag, coupling the common interface layer to every concrete implementation.
Each new backend requires editing the switch logic and shared dispatch functions — the opposite of LibJuno's goal of making components easy to adapt, share, and compose.
With the vtable pattern, a new implementation defines a new derived type plus a new static API table, without requiring edits to a central dispatcher.

**Vtable indirection vs. switch dispatch.** A vtable call adds one pointer dereference and typically prevents cross-interface inlining.
A switch-based dispatcher can sometimes be inlined more aggressively.
LibJuno favors architectural decoupling, substitutability, and test injection over dispatch micro-optimization — consistent with its emphasis on portability, explicit interfaces, and reusable building blocks.

**Shared vtable per type vs. per-instance function pointers.** LibJuno stores `const API_T *ptApi` in the root — one pointer per instance, with the function table shared by all instances of the same implementation.
This is smaller and more consistent than per-instance function pointers, especially for embedded targets with many instances sharing the same behavior.

**`const` vtable and ROM placement.** The module macros define the root field as `const API_T *ptApi`, and implementations provide `static const` API tables.
This supports placing vtables in read-only storage/flash on embedded systems, reducing RAM pressure and preventing accidental mutation.

**Why not `_Generic` or macro-driven generics.** `_Generic` is selected at compile time from the static type at the call site; it does not provide runtime substitution through a common root pointer.
LibJuno's module system is explicitly built around runtime polymorphism and injected dependencies, so `_Generic` would supplement type ergonomics, not replace the architecture.

=== Module System Pattern: Root / Derivation / Union

Every LibJuno module follows a three-part structural pattern:

[source,c]
----
// 1. Root — common interface fields
struct MY_ROOT_TAG JUNO_MODULE_ROOT(MY_API_T,
    // additional root-level state
);

// 2. Derivation — embeds root as first member
struct MY_IMPL_TAG JUNO_MODULE_DERIVE(MY_ROOT_T,
    // implementation-specific state
);

// 3. Union — sized for the largest variant
union MY_MODULE_T JUNO_MODULE(MY_API_T, MY_ROOT_T,
    MY_IMPL_T tImpl;
);
----

=== Design Rationale: Root/Derivation/Union Layering

**Why the root/derivation split.** API functions accept `ROOT_T *` — the common base type.
A flat struct per implementation would require every call site to know the concrete type (breaking abstraction) or use `void *` casts (breaking type safety).
Because `JUNO_MODULE_SUPER` is always the first member, a `DERIVATION_T *` can be safely cast to `ROOT_T *` by the C standard (§6.7.2.1¶15).

**Why the union.** The union serves stack allocation sizing with type erasure.
LibJuno requires zero dynamic allocation, so the caller must allocate the module instance — but may not know which derivation will be initialized at runtime.
The union is sized to the largest derivation.
It also provides C-standard type-punning safety (§6.5.2.3): accessing the root member is always valid because the root is a common initial sequence shared by all members.

**Why first-member embedding over `container_of`.** Three reasons:

. **No `offsetof` dependency** — the cast is `(ROOT_T *)ptDerived`, zero arithmetic, zero macros.
. **No ownership inversion** — root and derivation are physically contiguous in a single allocation.
. **Determinism and locality** — same cache line, no pointer indirection.

== Data Design

=== Common Types

LibJuno defines a small set of common value types used across all modules:

**`JUNO_STATUS_T`** (`int32_t`) — the unified error code type.
Zero indicates success; non-zero values indicate specific failure conditions.

**Result types** (`JUNO_MODULE_RESULT(NAME_T, OK_T)`) — structs combining `{JUNO_STATUS_T tStatus; OK_T tOk;}`.
Pre-defined specializations: `JUNO_RESULT_BOOL_T`, `JUNO_RESULT_UINT32_T`, `JUNO_RESULT_F64_T`, `JUNO_RESULT_SIZE_T`, `JUNO_RESULT_VOID_PTR_T`.

**Option types** (`JUNO_MODULE_OPTION(NAME_T, SOME_T)`) — structs combining `{bool bIsSome; SOME_T tSome;}`.
Used for operations where absence is a legitimate state (peek on empty collection, optional next-state in SM).

**`JUNO_POINTER_T`** — the fat pointer descriptor `{ptApi, pvAddr, zSize, zAlignment}`.
Carries self-describing type information and operations for type-safe memory access.

=== Memory Ownership Model

All memory in LibJuno is **caller-owned and injected**:

. The caller allocates storage (stack, static, or platform-specific memory pool).
. The caller passes storage to an `Init` function along with all dependencies.
. The module operates exclusively on the injected storage.
. The caller is responsible for the lifetime of all injected memory.

No module in LibJuno calls `malloc`, `calloc`, `realloc`, or `free`.
Memory pools (block allocator) operate on caller-provided backing buffers.

=== Design Rationale: Result/Option Types and Error Handling

**Why Rust-style Result/Option in C11.** `errno` is a hosted-only thread-local global unavailable in freestanding builds and violates the "no global mutable state" constraint.
Out-parameters separate the value from its validity — nothing prevents ignoring the status and using a garbage out-parameter.
Result structs bind the value and status in a single aggregate with `JUNO_ASSERT_OK` enforcing check-before-use.

**Why failure handlers are diagnostic-only.** Three reasons:

. **Predictable WCET** — if handlers could `longjmp` or abort, the control flow of every library function becomes unanalyzable.
. **No stack unwinding** — `longjmp`/`setjmp` is hosted-only; exceptions require RTTI. The only portable freestanding flow control is "check status, return status."
. **Separation of policy from mechanism** — the library decides what constitutes an error; the application decides what to do about it (log, blink LED, set fault flag).

**Why the ASSERT macro family.** Each macro maps to a specific domain of validation:

* `JUNO_ASSERT_EXISTS(ptr)` → always returns `JUNO_STATUS_NULLPTR_ERROR` — correct specific status.
* `JUNO_ASSERT_SUCCESS(s, ...)` → variadic action statements for ergonomic propagation.
* `JUNO_ASSERT_OK(result, ...)` → unwraps Result types.
* `JUNO_ASSERT_SOME(opt, ...)` → unwraps Option types.

Each macro name is self-documenting: it reads as a named proof obligation.

== Interface Design

=== Module Initialization Contracts

Every module follows the same initialization pattern:

. Caller allocates all storage.
. Caller calls `Module_Init(...)` passing the instance, all dependencies, and an optional failure handler.
. `Init` wires the vtable, stores dependencies, and calls `Verify`.
. `Verify` validates all pointers and dependencies are non-NULL.
. All subsequent public functions call `Verify` at entry.

[source,c]
----
JUNO_STATUS_T JunoDs_Heap_Init(
    JUNO_DS_HEAP_ROOT_T *ptHeap,
    const JUNO_DS_HEAP_POINTER_API_T *ptHeapPointerApi,
    JUNO_DS_ARRAY_ROOT_T *ptHeapArray,
    JUNO_FAILURE_HANDLER_T pfcnFailureHdlr,
    JUNO_USER_DATA_T *pvUserData
);
----

=== Vtable Dispatch Pattern

API dispatch follows this pattern:

[source,c]
----
// Caller invokes via vtable:
ptModule->ptApi->DoSomething(&ptModule->tRoot, args...);

// Or through the root directly:
JUNO_MODULE_GET_API(ptModule, ROOT_T)->DoSomething((ROOT_T *)ptModule, args...);
----

The vtable is `const` and typically `static` — enabling ROM placement.
Dispatch requires one pointer dereference through `ptApi`, then a function pointer call.

.Vtable Dispatch Sequence

=== Design Rationale: Pointer Trait System

**Why a fat pointer with a vtable.** `JUNO_POINTER_T` binds `{pvAddr, zSize, zAlignment, ptApi}` into a self-describing, self-validating value type.
In freestanding environments, `memcpy` is not guaranteed to exist; by putting `Copy` behind a vtable in the pointer, the element knows how to copy itself — data structures never need `memcpy`.

**Why Copy/Reset on the pointer, not per-data-structure.** One Copy/Reset implementation per element type, reused by every data structure (heap, queue, stack, map).
Data structures remain type-agnostic; adding a new element type requires only a new `JUNO_POINTER_API_T` implementation.

**Why `JUNO_TRAIT_ROOT` (no failure handler).** The pointer is a pass-by-value ephemeral descriptor, not a stateful module instance.
Adding failure handler fields (+16 bytes on 64-bit) would bloat every data structure call's stack frame.
Failure reporting belongs to the owning module.

=== Software Bus Integration Pattern

The broker provides single-threaded publish/subscribe message fan-out:

[source,c]
----
// Publisher:
ptBrokerApi->Publish(ptBroker, ENGINE_CMD_MID, tMsgPointer);

// Subscriber (in its own OnProcess):
ptQueueApi->Dequeue(&ptCmdPipe->tRoot, tReturnPointer);
----

=== Design Rationale: Single-Threaded Broker

**Why single-threaded scope.** IPC mechanisms are platform-specific and non-abstractable without losing the properties that make them useful (blocking semantics, copy vs. zero-copy, priority inheritance, WCET guarantees).
The broker solves intra-thread fan-out portably; inter-thread transport belongs to the integrator.

**Why pipes (queues) over callbacks.** Three reasons:

. **Temporal decoupling** — Publish does O(1) enqueue; subscriber processes on its own schedule.
. **Bounded backpressure** — each pipe has fixed capacity; overflow returns an error status.
. **Subscriber control** — subscribers inspect, batch, and prioritize message consumption.

**Multi-threaded integration pattern.** One broker per thread, bridged by a user-written bridge app that forwards messages between brokers via platform IPC.

=== Scheduler Integration Pattern

The scheduler is a table-driven cyclic executive:

[source,c]
----
JUNO_SCH_TABLE_NEW(ptTable, 2, 3,
    &tAppA.tRoot, &tAppB.tRoot,   // Minor frame 0
    &tAppA.tRoot, NULL,            // Minor frame 1
    &tAppC.tRoot, &tAppB.tRoot,   // Minor frame 2
);
----

=== Design Rationale: Cyclic Executive

**Why static cyclic executive.** Deterministic timing by construction — execution order and boundaries are fixed at compile time.
No OS dependency (works on bare metal with a single timer).
Certifiable by inspection (DO-178C, MISRA, IEC 61508).
Complements RTOS preemption rather than replacing it.

**Why flattened 2D array.** Zero allocation (single caller-owned buffer), cache-friendly sequential access, O(1) index arithmetic, NULL slots for variable-length frames.

**Why injected Time module.** Clock hardware (`Now`) and blocking primitives (`Sleep`, `SleepTo`) are platform-specific.
Time math (add, subtract, conversions) is universal and provided by LibJuno.
The split follows DI: inject what varies, provide what's universal.

== Requirements Allocation and Provenance

=== Requirements Allocation Strategy

LibJuno allocates requirements across design artifacts using the following rule:

* System-level requirements (`REQ-SYS-*`) are allocated to `docs/sdd/sdd.adoc`.
* Module-level requirements (`REQ-<MODULE>-*`) are allocated to `docs/sdd/modules/*.adoc`.
* Generated SRS and RTM artifacts are derived from `requirements/*/requirements.json` and traceability annotations in code, tests, and design files.

This allocation model preserves deterministic traceability from requirement IDs to design sections and avoids duplicate ownership of requirement statements.

=== Documentation Generation Provenance

The SRS and RTM are generated artifacts located under `docs/generated/`.
For release-quality outputs, regenerate documentation from current source annotations and requirements before publication.

[source,bash]
----
python3 scripts/generate_docs.py
----

Generated artifacts are treated as build outputs; the design source of record remains `docs/sdd/` and the requirement source of record remains `requirements/*/requirements.json`.

=== Full Module Status-Code Audit (Iteration 3)

A full pass was performed across `docs/sdd/modules/*.adoc` comparing documented status-code behavior to implementation evidence in:

* `src/*.c` runtime modules (array-dependent containers, memory, broker, time, crc/hash)
* Inline/`static` API logic in public headers under `include/juno/`
* Requirement-tagged unit tests in `tests/*.c`

Result summary:

* Confirmed and corrected mismatch: `docs/sdd/modules/software_bus.adoc` previously documented broker registry-full as `JUNO_STATUS_TABLE_FULL_ERROR`; implementation in `src/juno_broker.c` returns `JUNO_STATUS_ERR`.
* Previously corrected mismatch retained: `docs/sdd/modules/memory.adoc` zero-size `Get` uses `JUNO_STATUS_INVALID_SIZE_ERROR` per `src/juno_memory_block.c`.
* Remaining status-code wording reviewed in this pass was consistent with available source/test evidence.

== Public Header Reference

The following table lists all public C headers shipped with LibJuno and their purpose.
C++11 wrapper headers (`.hpp`) are included for completeness but are not covered in the detailed design sections of this SDD.

[cols="2,1,3"]
|===
| Header | Subsystem | Description

| `juno/status.h` | Core | Status codes, failure handler type, `JUNO_FAIL*` macros
| `juno/module.h` | Core | Module root/derivation/union macros, Result/Option types
| `juno/macros.h` | Core | `JUNO_ASSERT_EXISTS`, `JUNO_ASSERT_SUCCESS` assertion macros
| `juno/types.h` | Core | Pre-defined Result specializations (`JUNO_RESULT_BOOL_T`, etc.)
| `juno/memory/pointer_api.h` | Memory | Fat pointer trait, `JUNO_POINTER_T`, Copy/Reset API
| `juno/memory/memory_api.h` | Memory | Abstract allocator interface (`Get`/`Put`/`Update`)
| `juno/memory/memory_block.h` | Memory | Block allocator derivation with fixed-size pools
| `juno/ds/array_api.h` | Data Structures | Fixed-capacity array (`SetAt`/`GetAt`/`RemoveAt`)
| `juno/ds/heap_api.h` | Data Structures | Binary heap with injected Compare/Swap
| `juno/ds/queue_api.h` | Data Structures | Circular-buffer FIFO queue
| `juno/ds/stack_api.h` | Data Structures | LIFO stack
| `juno/ds/map_api.h` | Data Structures | Open-addressed hash map with linear probing
| `juno/crc/crc.h` | Algorithms | CRC-16/32 streaming update functions (ARC, BinHex, CCITT, CCITT32, Kermit, ZIP)
| `juno/hash/hash_djb2.h` | Algorithms | djb2 hash function
| `juno/math/juno_math.h` | Algorithms | Generic math macros (Abs, Max, Min, Clamp)
| `juno/math/juno_vec.h` | Algorithms | Vector operations (Add, Sub, Scale, Dot, Cross, Norm, Normalize)
| `juno/math/juno_vec_types.h` | Algorithms | Vector and quaternion type definitions
| `juno/math/juno_math_types.h` | Algorithms | Scalar type aliases (F32, F64)
| `juno/time/time_api.h` | System Services | Time interface, timestamp type, math and conversion functions
| `juno/io/async_io_api.h` | System Services | Asynchronous byte-stream IO interface
| `juno/io/i2c_io_api.h` | System Services | I2C bus transfer interface
| `juno/io/spi_io_api.h` | System Services | SPI transaction interface
| `juno/log/log_api.h` | System Services | Logging facade (Debug/Info/Warning/Error)
| `juno/mp/mtx_api.h` | System Services | Mutex interface (TryLock/Lock/Free)
| `juno/sm/sm_api.h` | Frameworks | State machine with explicit state transitions
| `juno/sb/broker_api.h` | Frameworks | Software Bus broker (Publish/RegisterSubscriber)
| `juno/app/app_api.h` | Frameworks | Application lifecycle (OnStart/OnProcess/OnExit)
| `juno/sch/juno_sch_api.h` | Frameworks | Cyclic executive scheduler
| `juno/module.hpp` | C++ Wrappers | C++11 module system helpers
| `juno/types.hpp` | C++ Wrappers | C++11 type wrappers
| `juno/ds/buff_api.hpp` | C++ Wrappers | C++11 queue buffer template
| `juno/ds/juno_buff.hpp` | C++ Wrappers | C++11 buffer implementation
|===
