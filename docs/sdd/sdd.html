1. Introduction

1.1. Purpose

This Software Design Document (SDD) describes the internal design of LibJuno version 1.0.1, a freestanding C11 embedded systems micro-framework. It follows the IEEE 1016 standard structure and is intended for developers, integrators, reviewers, and certification auditors who need to understand the design decisions, data structures, interfaces, and algorithms within the library.

1.2. Scope

This SDD covers all modules of LibJuno:

  • Core infrastructure: Status, Module System, Macros, Common Types

  • Memory subsystem: Pointer, Memory Allocator, Block Allocator

  • Data structures: Array, Heap, Queue, Stack, Map

  • Algorithms: CRC, Hash, Math (vectors, quaternions)

  • System services: Time, IO (Async, I2C, SPI), Log, Mutex

  • Frameworks: State Machine, Software Bus Broker, Application, Scheduler

2. System Architecture

2.1. Architectural Overview

LibJuno is a dependency-injection-oriented C11 micro-framework for embedded systems. Its architecture is built on four foundational principles:

  1. Zero dynamic allocation — all memory is caller-owned and injected.

  2. Freestanding compatibility — the library compiles with -nostdlib -ffreestanding.

  3. Vtable-based dependency injection — every module uses function-pointer tables for polymorphic dispatch.

  4. Explicit error propagation — all fallible functions return status codes; no exceptions, no errno, no longjmp.

The library provides reusable building blocks (data structures, memory management, CRC, math, time, IO abstractions) and a lightweight application framework (app lifecycle, state machine, software bus, scheduler) that integrators compose into their system.

2.2. Design Rationale: Why Vtable-Based DI

LibJuno chose vtable-based dependency injection over alternative C polymorphism approaches (switch-based dispatch, _Generic, per-instance function pointers) for the following reasons:

Architectural decoupling. A root object carrying a const API_T *ptApi pointer supports passing modules behind a stable root type. Dependencies can be injected explicitly, and concrete implementations can be swapped without central registries or framework-owned dispatch code.

Why not switch-based dispatch. A switch-based approach centralizes polymorphism around a type tag, coupling the common interface layer to every concrete implementation. Each new backend requires editing the switch logic and shared dispatch functions — the opposite of LibJuno’s goal of making components easy to adapt, share, and compose. With the vtable pattern, a new implementation defines a new derived type plus a new static API table, without requiring edits to a central dispatcher.

Vtable indirection vs. switch dispatch. A vtable call adds one pointer dereference and typically prevents cross-interface inlining. A switch-based dispatcher can sometimes be inlined more aggressively. LibJuno favors architectural decoupling, substitutability, and test injection over dispatch micro-optimization — consistent with its emphasis on portability, explicit interfaces, and reusable building blocks.

Shared vtable per type vs. per-instance function pointers. LibJuno stores const API_T *ptApi in the root — one pointer per instance, with the function table shared by all instances of the same implementation. This is smaller and more consistent than per-instance function pointers, especially for embedded targets with many instances sharing the same behavior.

const vtable and ROM placement. The module macros define the root field as const API_T *ptApi, and implementations provide static const API tables. This supports placing vtables in read-only storage/flash on embedded systems, reducing RAM pressure and preventing accidental mutation.

Why not _Generic or macro-driven generics. _Generic is selected at compile time from the static type at the call site; it does not provide runtime substitution through a common root pointer. LibJuno’s module system is explicitly built around runtime polymorphism and injected dependencies, so _Generic would supplement type ergonomics, not replace the architecture.

2.3. Module System Pattern: Root / Derivation / Union

Every LibJuno module follows a three-part structural pattern:

// 1. Root — common interface fields
struct MY_ROOT_TAG JUNO_MODULE_ROOT(MY_API_T,
    // additional root-level state
);

// 2. Derivation — embeds root as first member
struct MY_IMPL_TAG JUNO_MODULE_DERIVE(MY_ROOT_T,
    // implementation-specific state
);

// 3. Union — sized for the largest variant
union MY_MODULE_T JUNO_MODULE(MY_API_T, MY_ROOT_T,
    MY_IMPL_T tImpl;
);

2.4. Design Rationale: Root/Derivation/Union Layering

Why the root/derivation split. API functions accept ROOT_T * — the common base type. A flat struct per implementation would require every call site to know the concrete type (breaking abstraction) or use void * casts (breaking type safety). Because JUNO_MODULE_SUPER is always the first member, a DERIVATION_T * can be safely cast to ROOT_T * by the C standard (§6.7.2.1¶15).

Why the union. The union serves stack allocation sizing with type erasure. LibJuno requires zero dynamic allocation, so the caller must allocate the module instance — but may not know which derivation will be initialized at runtime. The union is sized to the largest derivation. It also provides C-standard type-punning safety (§6.5.2.3): accessing the root member is always valid because the root is a common initial sequence shared by all members.

Why first-member embedding over container_of. Three reasons:

  1. No offsetof dependency — the cast is (ROOT_T *)ptDerived, zero arithmetic, zero macros.

  2. No ownership inversion — root and derivation are physically contiguous in a single allocation.

  3. Determinism and locality — same cache line, no pointer indirection.

4.3. Interface Design

JUNO_FAIL(tStatus, pfcnFailureHandler, pvFailureUserData, pcMessage)

Macro that invokes the failure handler (if non-NULL) with the status, message, and user data. Does not alter control flow — callers must still return or handle the error.

JUNO_FAIL_MODULE(tStatus, ptMod, pcMessage)

Macro that invokes a module instance’s failure handler via JUNO_MODULE_SUPER fields. Null-safe: does nothing if ptMod or its handler is NULL.

JUNO_FAIL_ROOT(tStatus, ptMod, pcMessage)

Macro that invokes a module root’s failure handler directly. Null-safe: does nothing if ptMod or its handler is NULL.

4.4. Algorithm Descriptions

No algorithms. All operations are macro-based conditional invocations with O(1) cost.

4.5. Error Handling

The Status module itself defines the error-handling primitives used elsewhere. The JUNO_FAIL* macros guard against NULL before calling the handler, preventing null dereferences. The failure handler is diagnostic-only; it has no return value and does not affect the caller’s control flow.

4.6. Design Rationale

Status codes are int32_t rather than an enum to allow extensibility beyond the pre-defined set while maintaining a fixed-size ABI across compilation units. Failure handlers are callback-based (not abort or longjmp) to ensure deterministic control flow suitable for safety-critical embedded systems.

5. Module System

5.1. Purpose

The Module System (module.h) provides the structural primitives for vtable-based dependency injection in LibJuno. It defines macros for declaring module roots, derivations, and unions, as well as Result and Option wrapper types for deterministic error propagation.

5.2. Data Structures

5.2.1. Module Root (JUNO_MODULE_ROOT)

Expands to a struct body with fields:

const API_T *ptApi;                    // Vtable pointer (shared, const)
JUNO_FAILURE_HANDLER_T _pfcnFailureHandler;  // Optional failure callback
JUNO_USER_DATA_T *_pvFailureUserData;        // Opaque user data for handler
// ... caller-specified additional members

5.2.2. Module Derivation (JUNO_MODULE_DERIVE)

Expands to a struct body whose first member is the root:

ROOT_T tRoot;   // Embedded root (via JUNO_MODULE_SUPER)
// ... implementation-specific members

The first-member embedding guarantees that casting DERIVATION_T * to ROOT_T * is valid per C11 §6.7.2.1¶15.

5.2.3. Module Union (JUNO_MODULE)

Expands to a union containing the root and all derivations:

union {
    ROOT_T tRoot;           // Common view (via JUNO_MODULE_SUPER)
    DERIVATION_1_T tImpl1;  // Sized for this variant
    DERIVATION_2_T tImpl2;  // Sized for this variant
};

The union is sized to the largest derivation, enabling caller-side stack allocation without knowing the concrete type at runtime.

5.2.4. Trait Root (JUNO_TRAIT_ROOT)

A lightweight variant of JUNO_MODULE_ROOT that omits the failure handler fields:

const API_T *ptApi;
// ... trait-specific members (no failure handler)

Used for ephemeral, pass-by-value types like JUNO_POINTER_T.

5.2.5. Result Types (JUNO_MODULE_RESULT)

6. Macros Module

6.1. Purpose

The Macros module (macros.h) provides lightweight, freestanding-friendly assertion macros used throughout LibJuno. They return error codes rather than aborting, supporting deterministic error propagation without exceptions or longjmp.

6.2. Data Structures

No standalone data structures. The macros operate on JUNO_STATUS_T values and module pointers defined by the Status and Module System modules.

6.3. Interface Design

JUNO_ASSERT_EXISTS(ptr)

Returns JUNO_STATUS_NULLPTR_ERROR if the expression ptr evaluates to false (NULL, zero). Intended for validating required pointers within functions that return JUNO_STATUS_T.

JUNO_ASSERT_EXISTS_MODULE(ptr, ptMod, str)

Like JUNO_ASSERT_EXISTS but also invokes the module’s failure handler via JUNO_FAIL_MODULE before returning. Provides diagnostic context via the str message.

JUNO_ASSERT_SUCCESS(tStatus, …​)

Executes the variadic statements (typically return tStatus) if tStatus != JUNO_STATUS_SUCCESS. Used for ergonomic error propagation in call chains.

6.4. Algorithm Descriptions

No algorithms. All macros are O(1) conditional checks that expand inline.

6.5. Error Handling

Each macro is itself an error-handling primitive:

  • JUNO_ASSERT_EXISTS maps NULL pointers to JUNO_STATUS_NULLPTR_ERROR by construction.

  • JUNO_ASSERT_SUCCESS propagates any non-success status using caller-supplied actions.

  • JUNO_ASSERT_EXISTS_MODULE combines null-checking with diagnostic notification.

The macro names are self-documenting proof obligations: ASSERT_EXISTS reads as "this pointer must exist."

6.6. Design Rationale

The ASSERT macro family was chosen over raw if checks for three reasons:

  1. Correct status by construction — each macro maps to a specific JUNO_STATUS_T code, preventing mismatched return values.

  2. Ergonomic variadic propagation — JUNO_ASSERT_SUCCESS(s, return s) is concise and prevents forgetting error checks.

  3. Self-documenting — reading JUNO_ASSERT_EXISTS(ptArray) clearly states the obligation without additional comments.

8. Pointer Module

8.1. Purpose

The Pointer module (pointer_api.h) defines the fat pointer trait (JUNO_POINTER_T) and associated APIs used by all data structures and memory modules in LibJuno. 8.2. Data Structures

8.2.1. JUNO_POINTER_T (Trait Root)

  • Reset: resets the pointee to a defined empty state (e.g., zero-init).

8.2.3. JUNO_VALUE_POINTER_API_T

Macro that constructs a JUNO_POINTER_T literal from an API, C type (for sizeof/alignof), and address.

Macro that asserts a pointer’s type and API, returning on failure.

Inline function that verifies Copy and Reset are non-NULL.

Inline function that verifies Equals is non-NULL.

Inline function that validates API, address, and size are non-NULL/non-zero, then delegates to JunoMemory_PointerApiVerify.

8.4. Algorithm Descriptions

No algorithms. All operations are O(1) verification checks or literal construction.

8.5. Error Handling

  • JunoMemory_PointerVerify returns JUNO_STATUS_NULLPTR_ERROR if API, address, or size is invalid.

  • JunoMemory_PointerVerifyType returns JUNO_STATUS_ERR on any mismatch (size, alignment, API identity, alignment of address).

  • No failure handler is invoked — the pointer is a trait root without failure handler fields.

8.6. Design Rationale

Fat pointer with vtable. JUNO_POINTER_T binds {pvAddr, zSize, zAlignment, ptApi} into a self-describing, self-validating value type. In freestanding environments, memcpy is not guaranteed; by placing Copy behind a vtable, the element knows how to copy itself — data structures never require memcpy.

Copy/Reset on the pointer, not per-data-structure. One Copy/Reset implementation per element type, reused by every data structure (Array, Heap, Queue, Stack, Map). Data structures remain type-agnostic; adding a new element type requires only a new JUNO_POINTER_API_T implementation.

JUNO_TRAIT_ROOT (no failure handler). The pointer is a pass-by-value ephemeral descriptor, not a stateful module instance. Adding failure handler fields (+16 bytes on 64-bit) would bloat every data structure call’s stack frame. Failure reporting belongs to the owning module.

9. Memory Module

9.1. Purpose

The Memory module (memory_api.h, memory_block.h) provides the abstract memory allocation interface and its fixed-size block allocator implementation. It enables deterministic, bounded-capacity memory management on caller-supplied backing buffers without dynamic allocation.

9.2. Data Structures

9.2.1. Abstract Allocator Root (JUNO_MEMORY_ALLOC_ROOT_T)

struct JUNO_MEMORY_ALLOC_ROOT_TAG JUNO_MODULE_ROOT(JUNO_MEMORY_ALLOC_API_T,
    const JUNO_POINTER_API_T *ptPointerApi;  // Dependent pointer operations
);

Follows the module root pattern with an additional dependency on the pointer API.

9.2.2. Allocator API (JUNO_MEMORY_ALLOC_API_T)

Function Description

Get(ptMem, zSize)

Allocate a region of at least zSize bytes. Returns JUNO_RESULT_POINTER_T.

Update(ptMem, ptMemory, zNewSize)

Adjust an existing allocation’s descriptor in-place.

Put(ptMem, pvAddr)

Free a previously allocated region. Clears descriptor on success.

9.2.3. Block Allocator Derivation (JUNO_MEMORY_ALLOC_BLOCK_T)

struct JUNO_MEMORY_ALLOC_BLOCK_TAG JUNO_MODULE_DERIVE(JUNO_MEMORY_ALLOC_ROOT_T,
    uint8_t *pvMemory;                        // Backing memory region
    JUNO_MEMORY_BLOCK_METADATA_T *ptMetadata; // Per-block free-list metadata
    size_t zTypeSize;                         // Element size per block
    size_t zAlignment;                        // Alignment per block
    size_t zLength;                           // Total blocks in pool
    size_t zUsed;                             // Blocks currently allocated
    size_t zFreed;                            // Entries on free stack
);

9.2.4. Block Metadata (JUNO_MEMORY_BLOCK_METADATA_T)

Each entry stores the address of a freed block; the free list is a stack in ptMetadata[0..zFreed).

9.3. Interface Design

JunoMemory_BlockInit(…​)

Initializes a block allocator over caller-supplied backing memory and metadata arrays. Validates all parameters (nulls, overflow of zTypeSize * zLength, base alignment). Wires the vtable into the root. Counters zUsed and zFreed start at 0.

JunoMemory_BlockGetT(ptBlkRoot, type)

Convenience macro that calls Get with sizeof(type).

JunoMemory_BlockPutT(ptBlkRoot, pPtr)

Convenience macro that calls Put on a pointer descriptor.

JunoMemory_AllocApiVerify(ptAllocApi)

Inline verification that all three API functions (Get, Put, Update) are non-NULL.

JunoMemory_AllocVerify(ptAlloc)

Inline verification of the allocator root, its API, and its dependent pointer API.

9.3.1. Storage Declaration Macros

JUNO_MEMORY_BLOCK(name, type, length)

Declares a static backing array of type[length].

JUNO_MEMORY_BLOCK_METADATA(name, length)

Declares a static metadata array of JUNO_MEMORY_BLOCK_METADATA_T[length].

9.4. Algorithm Descriptions

9.4.1. Block Get (Allocation)

  1. If a freed block exists (zFreed > 0): pop from free stack (LIFO), decrement zFreed.

  2. Otherwise: carve a new block from offset zUsed * zTypeSize if zUsed < zLength; increment zUsed.

  3. Reset the block via the pointer API before returning.

  4. On Reset failure, roll back and report the error.

Complexity: O(1).

9.4.2. Block Put (Deallocation)

  1. Verify the address belongs to the pool and is properly aligned.

  2. Reject double-frees.

  3. If the block is the most recently allocated (last in sequence): decrement zUsed directly.

  4. Otherwise: push the address onto the free stack; increment zFreed.

  5. Clear the caller’s descriptor (pvAddr = NULL, sizes zeroed).

Complexity: O(1).

9.4.3. Block Update

  1. Adjust the descriptor’s zSize in-place, up to zTypeSize.

  2. Reject requests exceeding zTypeSize. Memory is never moved.

Complexity: O(1).

9.5. Error Handling

  • Get: returns JUNO_STATUS_MEMALLOC_ERROR when the pool is full or size exceeds element size. Returns JUNO_STATUS_NULLPTR_ERROR for zero-size requests.

  • Put: returns JUNO_STATUS_MEMFREE_ERROR for unaligned, out-of-range, or double-freed addresses. Restores the descriptor on failure.

  • Update: returns JUNO_STATUS_MEMALLOC_ERROR if the new size exceeds element capacity.

  • All operations call Verify at entry.

9.6. Design Rationale

The abstract JUNO_MEMORY_ALLOC_API_T separates the allocation interface from the block allocator implementation, allowing future allocator strategies (slab, buddy) to be injected through the same root type. The block allocator provides O(1) deterministic operations suitable for hard real-time systems with fixed-capacity pools.

10.1. Purpose

The Array module (array_api.h) provides a minimal, fixed-capacity array abstraction used as the foundational storage layer by higher-level data structures (Heap, Queue, Stack, Map). It defines three core operations — SetAt, GetAt, RemoveAt — over caller-owned, contiguous storage.

The Array module is deliberately thin — it provides only indexed access without length tracking, ordering, or growth semantics. Higher-level data structures (Heap, Queue, Stack, Map) compose the Array for storage and add their own invariants. This separation keeps the Array reusable and avoids duplicating contiguous-storage logic across multiple modules.

11.1. Purpose

11.3. Interface Design

11.4. Algorithm Descriptions

11.4.1. Insert

  1. Copy tValue into slot zLength via SetAt.

  2. Increment zLength.

  3. Bubble up: compare element with parent; swap if heap property violated; repeat until root or property holds.

11.4.2. Pop

  1. Copy root element (index 0) into tReturn via Copy.

  2. Swap root with last element (index zLength - 1).

  3. Remove last slot; decrement zLength.

  4. Sift down from index 0.

11.4.3. Heapify

  1. For i from (zLength / 2 - 1) down to 0: call SiftDown(i).

  2. This produces a valid heap in O(n) by bottom-up construction.

11.4.4. SiftDown

  1. Starting at iStart, compare with left and right children.

  2. Swap with the child that violates the heap property (if any).

  3. Repeat until no swap or leaf reached.

11.5. Error Handling

  • Insert returns JUNO_STATUS_ERR when the heap is full (zLength == capacity).

  • Pop returns JUNO_STATUS_ERR when empty (zLength == 0).

  • Heapify returns JUNO_STATUS_ERR when empty.

  • All operations propagate errors from Compare, Swap, SetAt, GetAt, and RemoveAt.

  • Verify is called at the start of every public function.

11.6. Design Rationale

The heap is parameterized by Compare/Swap rather than hardcoding a comparison operator, allowing the same heap implementation to serve max-heaps, min-heaps, and priority queues with arbitrary key types. The backing Array provides storage; the Pointer API provides element copying — keeping the heap logic type-agnostic.

12. Queue Module

12.1. Purpose

The Queue module (queue_api.h) provides a fixed-capacity FIFO queue built on the Array abstraction. Elements are stored in a circular buffer pattern with O(1) enqueue, dequeue, and peek operations.

12.2. Data Structures

12.2.1. Queue Root (JUNO_DS_QUEUE_ROOT_T)

struct JUNO_DS_QUEUE_ROOT_TAG JUNO_MODULE_ROOT(JUNO_DS_QUEUE_API_T,
    JUNO_DS_ARRAY_ROOT_T *ptQueueArray;  // Backing array
    size_t iStartIndex;                   // Index of logical front
    size_t zLength;                       // Current element count
);

12.2.2. Queue API (JUNO_DS_QUEUE_API_T)

Function Description

Enqueue(ptQueue, tItem)

Copy element into the back slot.

Dequeue(ptQueue, tReturn)

Copy front element out, advance front.

Peek(ptQueue)

Return pointer descriptor to the front element without removal.

12.3. Interface Design

JunoDs_QueueInit(ptQueue, ptQueueArray, pfcnFailureHdlr, pvFailureUserData)

Wires the queue vtable, stores the backing array, and validates via JunoDs_QueueVerify. iStartIndex and zLength are initialized to 0.

JunoDs_QueuePop(ptQueue, tReturn)

Implementation of Dequeue.

JunoDs_QueueApiVerify(ptQueueApi)

Inline verification that Enqueue, Dequeue, and Peek are non-NULL.

12.4. Algorithm Descriptions

12.4.1. Enqueue

  1. Compute back index: (iStartIndex + zLength) % capacity.

  2. Copy tItem into the backing array at that index via SetAt.

  3. Increment zLength.

12.4.2. Dequeue

  1. Copy the element at iStartIndex into tReturn via the pointer API.

  2. Remove the element at iStartIndex via RemoveAt.

  3. Advance iStartIndex = (iStartIndex + 1) % capacity.

  4. Decrement zLength.

12.4.3. Peek

  1. Return the pointer descriptor at iStartIndex via GetAt.

All operations are O(1).

12.5. Error Handling

  • Enqueue returns JUNO_STATUS_INVALID_SIZE_ERROR when full (zLength == capacity).

  • Dequeue returns JUNO_STATUS_ERR when empty (zLength == 0).

  • Peek returns JUNO_STATUS_INVALID_SIZE_ERROR when empty.

  • All operations call Verify at entry and propagate array/pointer errors.

12.6. Design Rationale

The Queue reuses the Array abstraction for storage, adding only circular-index arithmetic. This avoids duplicating contiguous-storage logic and benefits from the Array’s existing verification and pointer-based element access.

14.4. Algorithm Descriptions

14.4.1. Set (Insert/Update)

  1. Compute index = Hash(key) % capacity.

  2. Probe linearly: if slot is empty (IsValueNull returns true), copy entry via SetAt.

  3. If slot contains a matching key (Equals returns true), overwrite via SetAt.

  4. Continue probing with wrap-around until an empty slot or match is found.

  5. Return JUNO_STATUS_TABLE_FULL_ERROR if the probe traverses the full table.

14.4.2. Get (Lookup)

  1. Compute index = Hash(key) % capacity.

  2. Probe linearly: if slot matches key (Equals), return pointer descriptor via GetAt.

  3. If slot is empty (IsValueNull), key does not exist — return JUNO_STATUS_DNE_ERROR.

  4. If full traversal without match, return JUNO_STATUS_TABLE_FULL_ERROR.

14.4.3. Remove

  1. Compute index = Hash(key) % capacity.

  2. Probe linearly: if slot matches key, reset via RemoveAt and return success.

  3. If empty slot encountered before match, key was not present — return success (no-op).

All operations are O(1) average, O(n) worst-case under high load or clustered collisions.

14.5. Error Handling

  • Get returns JUNO_STATUS_DNE_ERROR when key not found.

  • Set returns JUNO_STATUS_TABLE_FULL_ERROR when no empty slot available.

  • Pointer type mismatches return JUNO_STATUS_INVALID_TYPE_ERROR.

  • All operations propagate errors from hashing, equality checks, and array access.

  • All operations call Verify at entry.

14.6. Design Rationale

Open addressing with linear probing was chosen for cache-friendliness and zero-allocation operation over a single contiguous array. The map is parameterized by three callback APIs (hash, equality, null-check) rather than hard-coding a specific key type, keeping it type-agnostic and reusable.

15.1. Purpose

The CRC module (crc.h) provides streaming update functions for six common CRC variants: ARC, BinHex, CCITT (16-bit), CCITT32 (32-bit), Kermit, and ZIP (CRC-32). All functions are pure, reentrant, and operate without module state or dynamic allocation.

15.3. Interface Design

All functions follow the same streaming pattern:

uint16_t Juno_CrcArcUpdate(uint16_t iCrc, const void *pcData, size_t zDataSize);
uint16_t Juno_CrcBinhexUpdate(uint16_t iCrc, const void *pcData, size_t zDataSize);
uint16_t Juno_CrcCcittUpdate(uint16_t iCrc, const void *pcData, size_t zDataSize);
uint32_t Juno_CrcCcitt32Update(uint32_t iCrc, const void *pcData, size_t zDataSize);

  • Input: running CRC (or seed for first chunk), data buffer, buffer length.

  • Output: updated CRC value.

  • NULL/zero: returns 0 when pcData is NULL or zDataSize is 0.

15.3.1. ZIP Streaming Note

Juno_CrcZipUpdate applies a final XOR (0xFFFFFFFF) before returning. For multi-chunk streaming, undo the XOR before passing it back: crc = Juno_CrcZipUpdate(crc ^ 0xFFFFFFFF, chunk, len).

15.4. Algorithm Descriptions

All CRC functions use table-driven computation:

  1. For each byte in the input buffer:

    1. Index into a 256-entry lookup table using the current CRC and the data byte.

    2. Update the running CRC with the table value and shift operations.

  2. Return the final CRC (with final XOR for ZIP).

The lookup tables are compiled into the library’s .c source files (juno_arc.c, juno_binhex.c, juno_ccitt.c, juno_ccitt32.c, juno_kermit.c, juno_zip.c).

Complexity: O(n) where n = zDataSize.

15.5. Error Handling

  • NULL data pointer or zero size: returns 0 (intentional, not an error status).

  • No status codes returned — CRC functions return the CRC value directly.

  • Functions are pure and reentrant — no global state.

15.6. Design Rationale

CRC functions are standalone (no module root or vtable) because they are stateless, pure computations with no dependencies to inject. The streaming update pattern allows incremental CRC calculation over multiple chunks without buffering the full message. Multiple CRC variants are provided because embedded protocols commonly require specific polynomials (CCITT for XMODEM, ZIP for file integrity, Kermit for serial protocols).

16. Hash Module

16.1. Purpose

The Hash module (hash_djb2.h) provides the djb2 hash function, a simple and effective hash algorithm used by the Map module for key hashing.

16.2. Data Structures

No module types. The function operates on byte buffers and returns JUNO_RESULT_SIZE_T.

16.3. Interface Design

JunoHash_Djb2(pcBuff, zBuffSize)

Inline function that computes the djb2 hash over zBuffSize bytes starting at pcBuff.

  • Returns {JUNO_STATUS_SUCCESS, hash_value} on success.

  • Returns {JUNO_STATUS_NULLPTR_ERROR, 0} if pcBuff is NULL.

16.4. Algorithm Descriptions

16.4.1. djb2

size_t zHash = 5381;
for (size_t i = 0; i < zBuffSize; i++) {
    zHash = ((zHash << 5) + zHash) + pcBuff[i];  // zHash * 33 + byte
}

The recurrence h = h * 33 + x with seed 5381 was proposed by Daniel J. Bernstein. It provides good distribution for typical key data (strings, byte sequences) with minimal computation.

Complexity: O(n) where n = zBuffSize.

16.5. Error Handling

  • NULL buffer: returns JUNO_STATUS_NULLPTR_ERROR without accessing memory.

  • Zero-length buffer: returns the seed value (5381) as the hash.

16.6. Design Rationale

djb2 was selected for its simplicity, small code size, and adequate distribution for hash-table usage in embedded systems. The function is static inline in the header to avoid link-time overhead and enable inlining by the compiler. It returns a JUNO_RESULT_SIZE_T (rather than a raw size_t) to maintain the library’s structural error-checking convention.

18.4. Algorithm Descriptions

18.4.1. AddTime

  1. Add subseconds: carry = (result overflows uint32_t).

  2. Add seconds with carry.

  3. No overflow detection on seconds (wraps at UINT32_MAX).

18.4.2. SubtractTime

  1. If subtraction would underflow: saturate to {0, 0}, return JUNO_STATUS_INVALID_DATA_ERROR.

  2. Otherwise: subtract subseconds with borrow, subtract seconds.

18.4.3. TimestampToNanos

  1. Compute iSeconds * 1e9 + (iSubSeconds * 1e9) / giSUBSECS_MAX.

  2. Round to nearest (half-up) for the fractional component.

  3. Detect overflow on the uint64_t result.

18.4.4. DoubleToTimestamp

  1. Integer part → iSeconds.

  2. Fractional part → iSubSeconds = (uint32_t)(frac * giSUBSECS_MAX).

  3. If frac >= 1 - 1/giSUBSECS_MAX: round up to next whole second.

All operations are O(1).

18.5. Error Handling

  • SubtractTime: returns JUNO_STATUS_INVALID_DATA_ERROR and saturates to zero on underflow.

  • Integer conversions: return JUNO_STATUS_INVALID_DATA_ERROR on overflow.

  • All functions validate the module pointer via JUNO_ASSERT_EXISTS.

18.6. Design Rationale

The time module splits platform-specific operations (Now, Sleep, SleepTo) from universal time math (add, subtract, conversions) via DI. Users inject their hardware clock through JunoTime_TimeApiInit and get all math for free. This follows the same DI principle: inject what varies, provide what’s universal.

The fixed-point subsecond representation was chosen over floating-point for deterministic arithmetic on targets without an FPU. The full-scale divisor (giSUBSECS_MAX) provides maximum fractional resolution within a 32-bit field.

22.4. Algorithm Descriptions

22.4.1. State Machine Execution Loop (User-Driven)

The SM module does not enforce an execution loop — the user drives it:

  1. Call GetCurrentState to get the active state.

  2. Call StateAction on the state.

  3. Call ShouldExit to check for transition.

  4. If exiting: call TransitionState to move to the next state.

  5. If next state is SOME: call ResetState on the new state.

This gives the user full control over when and how often to step the machine.

22.4.2. Transition

  1. Read ptCurrentState→tOptionNextState.

  2. If bIsSome: set ptSmRoot→ptCurrentState = tSome. Return SOME(newState).

  3. If not: return NONE. No change.

22.5. Error Handling

  • JunoSm_Init returns JUNO_STATUS_NULLPTR_ERROR if the SM root or start state is NULL.

  • JunoSm_StateInit returns JUNO_STATUS_NULLPTR_ERROR if required parameters are NULL.

  • JunoSm_GetCurrentState propagates Verify errors.

  • JunoSm_TransitionState propagates Verify errors. Returns NONE option (not error) when no next state is set.

  • All functions use JUNO_ASSERT_EXISTS for precondition checking.

22.6. Design Rationale

The SM is deliberately minimal — it provides state storage, transitions, and verification but does not enforce an execution model. The user calls StateAction, ShouldExit, and TransitionState explicitly, allowing integration into any scheduling scheme (cyclic executive, event-driven, RTOS task). States are full module roots with their own failure handlers, enabling per-state error reporting.

23. Software Bus (Broker) Module

23.1. Purpose

The Software Bus module (broker_api.h) provides a single-threaded publish/subscribe message broker. It solves intra-thread message fan-out: "A message arrived in my thread — who needs a copy?" The broker maintains a registry of subscriber pipes (queues) keyed by Message ID (MID) and fans out published messages to all matching pipes.

23.2. Data Structures

23.2.1. Broker Root (JUNO_SB_BROKER_ROOT_T)

A pipe is a queue derivation carrying a topic identifier. It inherits all queue operations (Enqueue, Dequeue, Peek) via the queue root.

uint32_t topic identifier used to route messages to matching pipes.

Function Description

Enqueue tMsg into all pipes matching tMid.

Add a pipe to the registry.

23.3. Interface Design

JunoSb_BrokerInit(ptBroker, ptPipeRegistry, iRegistryCapacity, pfcnFailureHdlr, pvFailureUserData)

Initializes the broker with a caller-owned pipe registry array. zRegistryLength starts at 0.

Inline function that sets the pipe’s MID and delegates to JunoDs_QueueInit for the underlying queue.

JunoSb_PipeVerify(ptPipe)

Validates the pipe and delegates to JunoDs_QueueVerify for the underlying queue.

23.4. Algorithm Descriptions

23.4.1. Publish

  1. For each pipe in the registry (0..zRegistryLength):

    1. If pipe→iMsgId == tMid: enqueue tMsg into the pipe via the queue API.

  2. Return status (propagate first error encountered).

Complexity: O(k) where k = registry length.

23.4.2. RegisterSubscriber

  1. Check capacity: if zRegistryLength >= zRegistryCapacity, return JUNO_STATUS_TABLE_FULL_ERROR.

  2. Store pipe pointer in ptPipeRegistry[zRegistryLength].

  3. Increment zRegistryLength.

Complexity: O(1).

23.5. Error Handling

  • Publish propagates queue enqueue errors (e.g., pipe full → JUNO_STATUS_INVALID_SIZE_ERROR).

  • RegisterSubscriber returns JUNO_STATUS_TABLE_FULL_ERROR when the registry is full.

  • All operations call Verify at entry.

  • JunoSb_PipeInit propagates JunoDs_QueueInit errors.

23.6. Design Rationale

Single-threaded by design. IPC mechanisms are platform-specific and non-abstractable without losing the properties that make them useful (blocking semantics, copy vs. zero-copy, priority inheritance, WCET guarantees). The broker solves intra-thread fan-out portably; inter-thread transport belongs to the integrator.

Pipes (queues) over callbacks. Three reasons:

  1. Temporal decoupling — Publish does O(1) enqueue per pipe; subscribers process on their own schedule.

  2. Bounded backpressure — each pipe has fixed capacity; overflow returns an error status.

  3. Subscriber control — subscribers inspect, batch, and prioritize message consumption.

Multi-threaded integration pattern. One broker per thread, bridged by a user-written bridge app that forwards messages between brokers via platform IPC. The broker is a routing table, not a transport.

25. Scheduler Module

25.1. Purpose

The Scheduler module (juno_sch_api.h) provides a table-driven cyclic executive that executes application OnProcess hooks in minor frames of fixed duration. It is the top-level framework component that composes Applications, Time, and the module system into a deterministic execution schedule.

25.2. Data Structures

25.2.1. Scheduler Root (JUNO_SCH_ROOT_T)

struct JUNO_SCH_ROOT_TAG JUNO_MODULE_ROOT(JUNO_SCH_API_T,
    JUNO_TIME_ROOT_T *ptTime;            // Injected time source
    JUNO_APP_ROOT_T **ptArrSchTable;     // Flattened 2D schedule table
    size_t zAppsPerMinorFrame;            // Slots per minor frame
    size_t zNumMinorFrames;               // Minor frames per major frame
    JUNO_TIMESTAMP_T tMinorFramePeriod;   // Duration of each minor frame
);

25.2.2. Scheduler API (JUNO_SCH_API_T)

Function Description

Execute(ptJunoSch)

Run the scheduler for one major frame.

GetMinorFramePeriod(ptJunoSch)

Return the configured minor frame period.

GetMajorFramePeriod(ptJunoSch)

Compute and return the major frame period.

25.2.3. Schedule Table Macro

JUNO_SCH_TABLE_NEW(ptArrName, zAppsPerMinorFrame, zNumMinorFrames, …​)

Declares a static array of JUNO_APP_ROOT_T * with the given dimensions. NULL entries indicate empty slots in variable-length frames.

25.3. Interface Design

The scheduler is initialized by populating a JUNO_SCH_ROOT_T with:

  1. A time module instance (for Now, SleepTo, time math).

  2. A flattened schedule table array.

  3. Frame dimensions and period.

Execute runs one major frame (all minor frames in sequence).

25.4. Algorithm Descriptions

25.4.1. Execute (One Major Frame)

tFrameStart = Now()
for each minor frame i in [0, zNumMinorFrames):
    tDeadline = tFrameStart + tMinorFramePeriod * (i + 1)
    for each slot j in [0, zAppsPerMinorFrame):
        ptApp = ptArrSchTable[i * zAppsPerMinorFrame + j]
        if ptApp != NULL:
            ptApp->ptApi->OnProcess(ptApp)
    SleepTo(tDeadline)

  1. Get the major frame start time via Now.

  2. For each minor frame, compute the deadline by adding (i + 1) * tMinorFramePeriod.

  3. Execute all non-NULL apps in the minor frame sequentially.

  4. Sleep until the minor frame deadline via SleepTo.

Complexity: O(F × A) per major frame, where F = minor frames and A = apps per frame.

25.4.2. GetMajorFramePeriod

Computes tMinorFramePeriod × zNumMinorFrames using AddTime.

25.4.3. Flattened 2D Indexing

The schedule table is a 1D array with index arithmetic:

\text{index}(i, j) = i \times \texttt{zAppsPerMinorFrame} + j

where $i$ is the minor frame index and $j$ is the app slot within the frame. NULL entries indicate unused slots, allowing variable-length minor frames within a fixed-width table.

25.5. Error Handling

  • Execute propagates errors from Now, AddTime, SleepTo, and each OnProcess call.

  • NULL app pointers in the table are silently skipped (not errors).

  • GetMinorFramePeriod and GetMajorFramePeriod return Result types with status codes.

25.6. Design Rationale

Static cyclic executive. Deterministic timing by construction — execution order and frame boundaries are fixed at compile time. No OS dependency (works on bare metal with a single timer). Certifiable by inspection (DO-178C, MISRA, IEC 61508). The scheduler complements RTOS preemption rather than replacing it — it fills the intra-thread scheduling gap.

Flattened 2D array. Zero dynamic allocation (single caller-owned buffer), cache-friendly sequential access, O(1) index arithmetic, NULL slots for variable-length frames without complex data structures.

Injected Time module. Clock hardware (Now) and blocking primitives (Sleep, SleepTo) are platform-specific. Time math (add, subtract, conversions) is universal and provided by LibJuno. The split follows the DI principle: inject what varies, provide what’s universal.

26. Data Design

26.1. Common Types

LibJuno defines a small set of common value types used across all modules:

JUNO_STATUS_T (int32_t) — the unified error code type. Zero indicates success; non-zero values indicate specific failure conditions.

Result types (JUNO_MODULE_RESULT(NAME_T, OK_T)) — structs combining {JUNO_STATUS_T tStatus; OK_T tOk;}. Pre-defined specializations: JUNO_RESULT_BOOL_T, JUNO_RESULT_UINT32_T, JUNO_RESULT_F64_T, JUNO_RESULT_SIZE_T, JUNO_RESULT_VOID_PTR_T.

Option types (JUNO_MODULE_OPTION(NAME_T, SOME_T)) — structs combining {bool bIsSome; SOME_T tSome;}. Used for operations where absence is a legitimate state (peek on empty collection, optional next-state in SM).

JUNO_POINTER_T — the fat pointer descriptor {ptApi, pvAddr, zSize, zAlignment}. Carries self-describing type information and operations for type-safe memory access.

26.2. Memory Ownership Model

All memory in LibJuno is caller-owned and injected:

  1. The caller allocates storage (stack, static, or platform-specific memory pool).

  2. The caller passes storage to an Init function along with all dependencies.

  3. The module operates exclusively on the injected storage.

  4. The caller is responsible for the lifetime of all injected memory.

No module in LibJuno calls malloc, calloc, realloc, or free. Memory pools (block allocator) operate on caller-provided backing buffers.

26.3. Design Rationale: Result/Option Types and Error Handling

Why Rust-style Result/Option in C11. errno is a hosted-only thread-local global unavailable in freestanding builds and violates the "no global mutable state" constraint. Out-parameters separate the value from its validity — nothing prevents ignoring the status and using a garbage out-parameter. Result structs bind the value and status in a single aggregate with JUNO_ASSERT_OK enforcing check-before-use.

Why failure handlers are diagnostic-only. Three reasons:

  1. Predictable WCET — if handlers could longjmp or abort, the control flow of every library function becomes unanalyzable.

  2. No stack unwinding — longjmp/setjmp is hosted-only; exceptions require RTTI. The only portable freestanding flow control is "check status, return status."

  3. Separation of policy from mechanism — the library decides what constitutes an error; the application decides what to do about it (log, blink LED, set fault flag).

Why the ASSERT macro family. Each macro maps to a specific domain of validation:

  • JUNO_ASSERT_EXISTS(ptr) → always returns JUNO_STATUS_NULLPTR_ERROR — correct specific status.

  • JUNO_ASSERT_SUCCESS(s, …​) → variadic action statements for ergonomic propagation.

  • JUNO_ASSERT_OK(result, …​) → unwraps Result types.

  • JUNO_ASSERT_SOME(opt, …​) → unwraps Option types.

Each macro name is self-documenting: it reads as a named proof obligation.

27. Interface Design

27.1. Module Initialization Contracts

Every module follows the same initialization pattern:

  1. Caller allocates all storage.

  2. Caller calls Module_Init(…​) passing the instance, all dependencies, and an optional failure handler.

  3. Init wires the vtable, stores dependencies, and calls Verify.

  4. Verify validates all pointers and dependencies are non-NULL.

  5. All subsequent public functions call Verify at entry.

JUNO_STATUS_T JunoDs_Heap_Init(
    JUNO_DS_HEAP_ROOT_T *ptHeap,
    const JUNO_DS_HEAP_POINTER_API_T *ptHeapPointerApi,
    JUNO_DS_ARRAY_ROOT_T *ptHeapArray,
    JUNO_FAILURE_HANDLER_T pfcnFailureHdlr,

);

27.2. Vtable Dispatch Pattern

API dispatch follows this pattern:

// Caller invokes via vtable:
ptModule->ptApi->DoSomething(&ptModule->tRoot, args...);

// Or through the root directly:
JUNO_MODULE_GET_API(ptModule, ROOT_T)->DoSomething((ROOT_T *)ptModule, args...);

The vtable is const and typically static — enabling ROM placement. Dispatch requires one pointer dereference through ptApi, then a function pointer call.

27.3. Trait System (JUNO_TRAIT_ROOT)

Traits are lightweight interfaces without failure handler fields:

struct JUNO_POINTER_TAG JUNO_TRAIT_ROOT(JUNO_POINTER_API_T,
    void *pvAddr;
    size_t zSize;
    size_t zAlignment;
);

27.4. Design Rationale: Pointer Trait System

Why a fat pointer with a vtable. JUNO_POINTER_T binds {pvAddr, zSize, zAlignment, ptApi} into a self-describing, self-validating value type. In freestanding environments, memcpy is not guaranteed to exist; by putting Copy behind a vtable in the pointer, the element knows how to copy itself — data structures never need memcpy.

Why Copy/Reset on the pointer, not per-data-structure. One Copy/Reset implementation per element type, reused by every data structure (heap, queue, stack, map). Data structures remain type-agnostic; adding a new element type requires only a new JUNO_POINTER_API_T implementation.

Why JUNO_TRAIT_ROOT (no failure handler). The pointer is a pass-by-value ephemeral descriptor, not a stateful module instance. Adding failure handler fields (+16 bytes on 64-bit) would bloat every data structure call’s stack frame. Failure reporting belongs to the owning module.

27.8. Design Rationale: Cyclic Executive

Why static cyclic executive. Deterministic timing by construction — execution order and boundaries are fixed at compile time. No OS dependency (works on bare metal with a single timer). Certifiable by inspection (DO-178C, MISRA, IEC 61508). Complements RTOS preemption rather than replacing it.

Why flattened 2D array. Zero allocation (single caller-owned buffer), cache-friendly sequential access, O(1) index arithmetic, NULL slots for variable-length frames.

Why injected Time module. Clock hardware (Now) and blocking primitives (Sleep, SleepTo) are platform-specific. Time math (add, subtract, conversions) is universal and provided by LibJuno. The split follows DI: inject what varies, provide what’s universal.
